# PM3 API 设计文档生成器

## 核心任务

将非结构化的技术需求描述转换为标准化的 PM3 API 契约文档，确保技术实现与业务需求一致，并严格遵循 PM3 项目的 API 设计规范。

---

## 工作流程

### 阶段 1：文档解析与信息提取

#### 1.1 读取引用文档

必须首先读取以下两个参考文档以理解 PM3 API 设计规范：

📄 20_项目/PM3/04-API/references/PM3-API 规则文档.md
📄 20_项目/PM3/04-API/references/PM3-API 模板文档.md

**从规则文档提取**：
- API 命名规范（URL 路径、大小写、资源单复数）
- HTTP 方法使用规则
- 权限标识命名规则
- 入参/出参结构规范
- 错误码定义标准
- DTO 命名规范

**从模板文档提取**：
- 接口详细模板结构
- 请求/响应示例格式
- 字段说明格式
- Controller 代码模板

#### 1.2 解析用户输入文档

读取用户提供的两份文档：

**技术需求分析文档**：
- 重点提取 Section 5（API 设计骨架）
- 识别 API 清单条目
- 提取业务场景描述

**数据库设计文档**：
- 提取实体关系（ER 图）
- 提取表结构定义
- 提取字段类型、约束、校验规则
- 识别主键、外键关系

### 阶段 2：需求映射与冲突检测

#### 2.1 API-数据库映射

为每个 API 条目建立与数据库实体的映射：

| API 条目 | 涉及表 | 数据操作 | 验证状态 |
|---------|--------|---------|---------|
| 用户列表 | sys_user | SELECT | ✅ 可行 |
| 角色详情 | sys_role | SELECT | ✅ 可行 |
| 权限分配 | sys_user_role | INSERT/DELETE | ✅ 可行 |

**验证要点**：
- API 所需字段是否在数据库表中存在
- 外键关系是否支持关联查询
- 是否需要新增字段或表

#### 2.2 冲突检测

检测以下潜在冲突并标注需人工确认：

⚠️ 命名风格冲突
- 需求中使用"user_list"，规则要求 camelCase → 需确认为"/user/list"

⚠️ HTTP 方法误用
- 需求中"删除用户"使用 GET → 应改为 POST/DELETE

⚠️ 缺失必填元素
- API 缺少权限标识 → 需补充"{module}:delete"

⚠️ 参数校验缺失
- 新增接口缺少@Valid 标注 → 需补充

### 阶段 3：接口详细设计

针对每个 API 条目，按以下结构生成详细设计：

#### 3.1 基础信息

```markdown
### [模块名称][操作类型][资源名称]

**接口路径**：`{METHOD} {基础路径}/{操作}`
**权限标识**：`{module}:{operation}`
**功能说明**：[一句话描述接口功能]
```

**命名规范检查**：
- ✅ 路径：`/api/{module}/{resource}/{operation}`（kebab-case 或 camelCase）
- ✅ 权限：模块：操作（根据模块类型选择命名风格）
- ✅ 方法：查询用 GET，增改用 POST/PUT，删除用 POST/DELETE

#### 3.2 请求参数设计

根据数据库字段定义生成请求参数表：

| 参数名 | 类型 | 位置 | 必填 | 默认值 | 校验规则 | 示例值 | 说明 |
|-------|------|------|------|--------|---------|--------|------|
| userId | String | Body | 是 | - | @NotBlank | "U001" | 用户 ID |
| userName | String | Body | 是 | - | @Size(max=50) | "张三" | 用户名称 |
| email | String | Body | 否 | - | @Email | "test@example.com" | 邮箱 |
| status | Integer | Body | 否 | 1 | @Min(0),@Max(1) | 1 | 状态 |

**参数位置规则**：
- Path Variable：资源标识符（ID、Code）
- Query Parameter：GET 请求的简单筛选条件
- Request Body：POST/PUT 请求的复杂参数（必须使用参数对象）

**校验规则映射**：
- 数据库 NOT NULL → @NotNull / @NotBlank
- VARCHAR(n) → @Size(max=n)
- 数值范围 → @Min / @Max
- 邮箱格式 → @Email
- 正则匹配 → @Pattern

#### 3.3 响应结构设计

根据数据库字段和 DTO 规范生成响应结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "userId": "U001",
    "userName": "张三",
    "email": "test@example.com",
    "status": 1,
    "createTime": "2024-01-15 10:30:00"
  }
}
```

**分页响应**（列表接口）：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "total": 100,
    "records": [...],
    "pageNum": 1,
    "pageSize": 10,
    "pages": 10
  }
}
```

**树形响应**（组织/菜单等）：

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "key": "1",
      "title": "研发部",
      "children": [],
      "isLeaf": false
    }
  ]
}
```

#### 3.4 异常场景与错误码

为每个接口定义错误响应：

| 错误码 | 错误信息 | 触发条件 | 处理建议 |
|-------|---------|---------|---------|
| 401 | 未授权 | Token 缺失或过期 | 重新登录 |
| 403 | 无权限 | 用户无该接口权限 | 申请权限 |
| 404 | 资源不存在 | 请求的 ID 不存在 | 检查 ID 参数 |
| 400 | 请求参数错误 | 参数校验失败 | 修正参数 |
| 500 | 系统异常 | 服务器内部错误 | 联系运维 |

#### 3.5 数据库操作关联

明确接口涉及的数据库操作：

**数据库操作**：
- 涉及表：`sys_user`, `sys_role`, `sys_user_role`
- 操作类型：`SELECT` / `INSERT` / `UPDATE` / `DELETE`
- 事务边界：[单表操作无需事务 / 多表操作需@Transactional]
- 性能注意事项：
  - 列表查询需添加分页，禁止全表扫描
  - 关联查询需使用索引字段
  - 批量操作建议使用批量 SQL

## 特殊情况处理

### API 清单不完整

当 Section 5 中 API 条目缺少字段定义时：

1. **基于数据库设计推断**：
   - 根据表结构推断必填字段（NOT NULL 字段）
   - 根据字段类型推断参数类型
   - 根据注释推断字段说明

2. **标注推断字段**：

   [推断] 字段"status"基于数据库默认值为 1 推断为可选参数
   [待确认] 字段"description"在需求中未提及，是否需要在 API 中暴露？
### 需求与规则冲突

当需求描述与规则文档冲突时：

1. **标记冲突点**：明确说明冲突内容
2. **提供建议**：基于 PM3 规范给出建议
3. **请求确认**：标注需人工决策的项

示例：

⚠️ 冲突检测：
需求中"删除用户"接口路径为`/user/deleteUser`（动词结尾）
PM3 规则要求资源名词结尾，建议改为`/user/delete`
