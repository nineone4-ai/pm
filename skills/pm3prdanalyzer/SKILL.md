## 定位说明

本 Skill 的使用者是**技术经理**，输出对象是**开发人员**。包含两个可独立执行的命令：

| 命令          | 触发词                                   | 说明                                                         |
| ------------- | ---------------------------------------- | ------------------------------------------------------------ |
| **CMD-INDEX** | `生成PRD索引` / `提取功能索引` / `index` | 将PRD功能性需求提炼为压缩版索引文档，后续分析直接引用，无需重复读原始PRD |

两个命令可**独立执行**，也可**顺序执行**：先运行 CMD-INDEX 生成索引，再运行 CMD-ANALYZE 基于索引进行分析，大幅降低后续 token 消耗。

## 入参解析（每次执行必须先完成此步骤）

Skill 接收两个位置参数，在执行任何命令前必须先确认这两个参数均已就绪：

| 参数               | 说明                             | 示例                  |
| ---------------- | ------------------------------ | ------------------- |
| `$1` **PRD文档路径** | 原始PRD文件的路径（绝对路径或相对路径均可）        | `/docs/pm3-开工申请.md` |
| `$2` **模块名称**    | 本次分析的目标模块名，对应PRD中的 `## {模块名称}` | `开工申请`              |

### 参数缺失时的提示规则

**规则1：两个参数均缺失**（用户直接输入触发词，未带任何参数）

立即以如下格式提示，不执行任何分析：

```
需要提供以下信息才能开始：

📄 PRD文档路径（参数1）：请输入PRD文件的路径
   示例：/workspace/docs/项目PRD.md

📦 模块名称（参数2）：请输入要分析的模块名称（对应PRD中的二级标题）
   示例：开工申请 / 费用报销 / 采购管理

用法示例：
  /prd-analyzer index /docs/项目PRD.md 开工申请
  /prd-analyzer analyze /docs/项目PRD.md 开工申请
```

**规则2：仅缺少 `$2` 模块名称**

读取PRD文档，扫描所有 `## {模块名称}` 二级标题，列出可选模块并提示用户选择：

```
已读取PRD：{$1路径}
发现以下模块，请指定要分析的模块名称：

  1. {模块名A}
  2. {模块名B}
  3. {模块名C}

请回复模块名称或序号，例如：开工申请 或 1
```

**规则3：两个参数均已提供**

直接执行对应命令，不再询问，首行输出确认信息：

```
▶ 执行：{CMD-INDEX / CMD-ANALYZE}
  PRD：{$1}
  模块：{$2}
```

## PRD 标准结构说明

```
PRD文档
├── 概述区（目的/范围/术语）
├── 背景区
├── 整体设计
│   ├── 功能清单                 ← Epic 识别源
│   ├── 角色定义                 ← 仅提取数据权限范围
│   └── 相关接口清单             ← 外部系统依赖
├── 功能性需求（核心）
│   └── ## {模块名称}            ← Epic 层级
│       ├── ### 业务流程
│       ├── ### 审批流程
│       ├── ### 功能方案
│       │   └── #### {功能点}   ← Story 识别源
│       ├── ### 使用范围
│       └── ### 功能逻辑         ← 索引提取核心区
│           └── #### {操作类型}
│               ├── ##### 功能描述
│               ├── ##### 按钮逻辑
│               ├── ##### 列表逻辑
│               ├── ##### 查询条件
│               ├── ##### 列表/详情
│               ├── ##### 状态说明
│               └── ##### 补充逻辑
└── 非功能性需求
```

## CMD-INDEX：生成PRD功能索引文档（地图+导航架构）

### 命令目标

将PRD「功能性需求」章节提炼为**两层架构**：
- **索引层（地图）**：功能全景，1页纸看完模块全貌
- **详情层（导航）**：按需深入，每个功能点独立文档

### 两层架构说明

```
CMD-INDEX 输出
├── 功能索引.md          ← 索引层（地图）：全景图 + 功能清单 + 跨功能关联
└── details/
    ├── {功能点A}.md      ← 详情层（导航）：字段、规则、状态机详情
    ├── {功能点B}.md
    └── ...
```

**索引层职责**（纯粹、轻薄）：
- 回答"我在哪、能去哪、怎么去"
- 不包含：字段定义、按钮权限、业务规则详情

**详情层职责**（完整、按需）：
- 回答"这个功能具体怎么做"
- 包含：字段清单、业务规则、状态机、审批动作

### 内容分层规则

| 内容 | 归属层 | 说明 |
|------|--------|------|
| 功能点名称 | 索引层 | 用于定位 |
| 一句话描述 | 索引层 | 快速理解功能 |
| 核心实体 | 索引层 | 关联数据模型 |
| 功能点间关系 | 索引层 | Mermaid 流程图 |
| 字段定义 | 详情层 | 类型/必填/联动 |
| 按钮权限 | 详情层 | 显示条件/校验 |
| 业务规则 | 详情层 | 公式/阈值/边界 |
| 状态机 | 索引层(简) + 详情层(全) | 索引层只列状态名，详情层有完整转换图 |

---

### 执行流程（扫描-决策-执行-质检）

```
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 扫描 (Scan)                                       │
│  └── 使用 Context-Mode 索引 PRD，提取结构元数据              │
├─────────────────────────────────────────────────────────────┤
│  Phase 2: 决策 (Decide)                                     │
│  └── 根据 PRD 大小选择处理策略                               │
├─────────────────────────────────────────────────────────────┤
│  Phase 3: 执行 (Execute)                                    │
│  ├── 策略A: Subagent 一次性处理（小PRD）                    │
│  └── 策略B: Context-Mode 分批处理（大PRD）                  │
├─────────────────────────────────────────────────────────────┤
│  Phase 4: 质检 (Quality Check)                              │
│  └── 抽样对比 PRD 原文，生成质量报告                          │
└─────────────────────────────────────────────────────────────┘
```

### Phase 1：扫描（Scan）

**目标**：快速了解 PRD 规模和结构，无需加载完整内容到上下文。

**执行步骤**：

**Step S-1：索引 PRD 文档**

使用 `ctx_index` 将 PRD 文档索引到 knowledge base：
```javascript
ctx_index({
  path: $1,  // PRD 文件路径
  source: `PRD:${$2}`  // 模块名作为 source 标记
})
```

**Step S-2：查询结构元数据**

使用 `ctx_search` 查询以下信息：

```
查询1: "有哪些 Epic 模块 二级标题"
查询2: "每个 Epic 下有哪些功能点 四级标题"
查询3: "有多少个功能点包含字段表格"
查询4: "有多少个功能点包含状态机描述"
查询5: "预估整个功能性需求章节有多少字"
```

**Step S-3：构建结构摘要**

整理为元数据对象：
```javascript
{
  prdPath: $1,
  moduleName: $2,
  epicCount: 3,           // Epic 数量
  featureCount: 12,       // 功能点总数
  complexFeatures: 5,     // 含复杂字段/规则的功能点数
  stateMachines: 2,       // 状态机数量
  estimatedTokens: 6000,  // 预估 token 数
  hasComplexTables: true  // 是否有复杂表格
}
```

---

### Phase 2：决策（Decide）

**目标**：根据结构摘要，选择最优处理策略。

**决策矩阵**：

| 维度 | 阈值 | 权重 |
|------|------|------|
| 功能点总数 | > 15 | 高 |
| 单功能点平均字段数 | > 10 | 高 |
| 预估 Token 数 | > 8000 | 高 |
| 状态机数量 | > 2 | 中 |
| 有复杂计算规则 | 是 | 中 |

**决策规则**：

```
IF (功能点 > 15) OR (预估Token > 8000) OR (复杂功能点 > 8)
THEN → 策略B: Context-Mode 分批处理
ELSE → 策略A: Subagent 一次性处理
```

**决策输出**：
```javascript
{
  strategy: "A" | "B",
  reason: "功能点过多(18个)，建议分批处理",
  estimatedTime: "2-3分钟" | "5-8分钟",
  batchSize: undefined | 3  // 策略B时每批处理功能点数
}
```

### Phase 3：执行（Execute）

根据 Phase 2 的决策结果，执行相应的处理策略。

**Step A-1：启动 Subagent**

    2. 按 Step I-1 ~ I-6 生成索引层和详情层

    - 详情层：references/detail-template.md
  `
})
```

**Step A-2：等待结果**

Subagent 完成以下工作：
- 读取完整 PRD
- 生成 `功能索引.md`
- 生成 `details/*.md`（按需）
- 返回执行报告

**Step A-3：展示结果**

输出执行报告：
```

#### 策略B：Context-Mode 分批处理（大PRD）

**适用场景**：功能点 ≥ 15，结构复杂，预估 token ≥ 8000

**执行步骤**：

**Step B-1：生成索引层（主线程）**

使用 `ctx_search` 查询必要信息，生成轻量级索引层：

```javascript
// 查询功能点清单（仅名称+描述）
ctx_search({
  queries: [
    "列出所有功能点的名称和一句话描述",
    "功能点之间的数据流转关系",
    "共享的实体有哪些"
  ],
  source: `PRD:${$2}`
})
```

基于查询结果，生成 `功能索引.md`（仅索引层，无详情）。

**Step B-2：分批生成功能点详情**

将功能点分组，每组 3-5 个（可配置）：

```javascript
batches = [
  ["功能点1", "功能点2", "功能点3"],
  ["功能点4", "功能点5", "功能点6"],
  // ...
]
```

对每批启动 Subagent（可并行）：

```javascript
for (batch of batches) {
  Agent({
    description: `生成功能点详情 Batch-${index}`,
    prompt: `
      请为以下功能点生成详情文件：${batch.join(", ")}

      使用 Context-Mode 查询 PRD：
      - source: PRD:${$2}
      - 查询每个功能点的：字段清单、业务规则、状态机

      输出到：details/{功能点名称}.md
    `
  })
}
```

**Step B-3：汇总结果**

等待所有 batch 完成，更新索引层的待确认项汇总。

**Step B-4：展示结果**

```
✅ CMD-INDEX 执行完成（策略B: Context-Mode分批）

PRD规模：18个功能点，预估9000 tokens
处理策略：分4批并行处理

生成文件：
- 功能索引.md (0.8KB) - 索引层
- details/功能点1.md (1.2KB)
- details/功能点2.md (0.9KB)
- ... (共18个详情文件)

待确认项：5项
- [ ] Batch1-功能点1: ...
- [ ] Batch3-功能点7: ...
```

### Phase 4：质量检查（Quality Check）【关键】

**目标**：确保生成的索引文档与原始PRD一致，无遗漏、无偏差、无错误。

**检查范围**：
- 索引层（功能索引.md）：功能点完整性、描述准确性
- 详情层（details/*.md）：字段完整性、规则准确性、状态机一致性

---

#### 检查策略

根据 Phase 2 的决策结果，选择检查策略：

```
IF 策略A (小PRD)
  └── 完整检查：逐个功能点详细对比

IF 策略B (大PRD)
  └── 抽样检查：每个 Batch 抽查 1-2 个功能点
  └── 关键路径全检：涉及核心业务流程的功能点必须检查
```

---

#### 检查维度与标准

| 检查项 | 检查内容 | 通过标准 | 权重 |
|--------|---------|---------|------|
| **字段完整性** | 字段名称、类型、必填、只读 | 与PRD字段表100%一致 | 🔴 高 |
| **枚举值准确性** | 下拉选项、状态值 | 枚举值顺序、名称完全一致 | 🔴 高 |
| **规则准确性** | 业务规则、计算公式 | 逻辑与PRD描述一致，阈值正确 | 🔴 高 |
| **状态机一致性** | 状态定义、转换路径 | 状态名、转换条件完全一致 | 🟡 中 |
| **按钮权限** | 显示条件、点击校验 | 条件逻辑正确，无遗漏 | 🟡 中 |
| **联动规则** | 字段显示/隐藏逻辑 | 联动条件和PRD一致 | 🟡 中 |
| **审批节点** | 特殊动作、数据影响 | 节点动作描述准确 | 🟡 中 |
| **关联流程** | 自动带出规则 | 触发条件、限制正确 | 🟢 低 |

---

#### 执行步骤

**Step Q-1：建立检查索引**

使用 Context-Mode 索引生成的文档：

```javascript
// 索引生成的功能索引文档
ctx_index({
  path: `/20_项目/PM3/01-requirements/${$2}/${prdName}_功能索引.md`,
  source: `INDEX:${$2}`
})

// 索引所有详情文件
for (detailFile of detailFiles) {
  ctx_index({
    path: detailFile,
    source: `DETAIL:${$2}`
  })
}
```

**Step Q-2：抽样选择检查点**

根据策略选择需要检查的功能点：

```javascript
// 策略A：检查所有功能点
checkPoints = allFeatures

// 策略B：抽样检查（每批1-2个）+ 关键路径全检
checkPoints = [
  ...sampleFromEachBatch(2),     // 每批抽2个
  ...criticalPathFeatures         // 核心业务流程全检
]
```

**Step Q-3：逐项对比检查**

对每个检查点，执行以下查询对比：

```javascript
// 查询1: PRD中的原始字段定义
prdFields = ctx_search({
  queries: [`${featureName} 字段清单 字段名 类型 必填`],
  source: `PRD:${$2}`
})

// 查询2: 生成的字段定义
generatedFields = ctx_search({
  queries: [`${featureName} 字段清单`],
  source: `DETAIL:${$2}`
})

// 对比：字段数量、名称、类型、必填是否一致
```

```javascript
// 查询3: PRD中的业务规则
prdRules = ctx_search({
  queries: [`${featureName} 业务规则 计算公式 阈值`],
  source: `PRD:${$2}`
})

// 查询4: 生成的业务规则
generatedRules = ctx_search({
  queries: [`${featureName} 业务规则`],
  source: `DETAIL:${$2}`
})

// 对比：规则逻辑、公式、数值是否一致
```

## 注意事项

1. **CMD-INDEX 的核心价值是去噪**：UI交互细节对开发几乎无用，全部去掉；字段和规则一个都不能漏
2. **业务规则不要概括归纳**：计算公式、阈值、边界条件必须完整保留原文，概括后容易失真
3. **审批/权限只声明不实现**：这是与通用PRD分析最大的区别
4. **特殊业务动作是重点**：审批节点上的字段填写、外部写入、条件分支必须单独拆 Task
5. **集成时机必须明确**：读写外部系统的触发时机（创建时/提交时/审批通过时）在 Task 中明确说明
6. **字段给出中英文对照**：英文命名用驼峰，供开发直接用于编码
